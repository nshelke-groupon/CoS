# Deployment

## Overview

All tdo-team components are deployed as Kubernetes CronJobs using a shared Docker image (`docker-conveyor.groupondev.com/tdo/tdo-team`). Deployment is driven by Helm 3 using the `cmf-generic-cron-job` chart (version 3.89.0) from Groupon's internal Helm registry. Each of the eight CronJob components is rendered from two YAML config files (common + per-environment) and deployed to Kubernetes namespaces via `krane`. CI/CD is managed by Jenkins using Groupon's shared `java-pipeline-dsl` library; builds trigger on `master` branch and auto-promote from staging to production.

## CI/CD Pipeline

- **Tool**: Jenkins (`java-pipeline-dsl@latest-2`)
- **Config**: `Jenkinsfile` at repo root
- **Trigger**: Push to `master` branch; `releasableBranches: ['master']`
- **Slack notifications**: `#reliability-devs` channel on start and any failure
- **Node label**: `dind_2gb_2cpu` (Docker-in-Docker build node)

### Pipeline Stages

1. **Checkout**: Clones repo including Git submodules (`cloneSubModules: true`)
2. **Docker Build**: Builds Docker image from repo root `Dockerfile`; tags with version
3. **Push**: Pushes image to `docker-conveyor.groupondev.com/tdo/tdo-team`
4. **Deploy to staging-us-central1**: Auto-deploys on successful build; runs `deploy.sh staging-us-central1 staging tdo-team-staging` via krane for all 8 CronJob components
5. **Promote to production-us-central1**: Auto-promotes from `staging-us-central1` (`promote_to: production-us-central1`)
6. **Deploy to production-us-west-1**: Manual or scheduled deploy; runs `deploy.sh production-us-west-1 production tdo-team-production`

### Deployment Script Flow

The `deploy.sh` script renders all 8 CronJob components via `helm3 template` and pipes the combined YAML into a single `krane deploy` call. This means all components are deployed atomically in one Kubernetes apply operation. The Helm set arguments `appVersion` and `changeCause` are injected from DeployBot environment variables (`DEPLOYBOT_PARSED_VERSION`, `DEPLOYBOT_LOGBOOK_TICKET`).

## Health Checks

All CronJob pods use exec probes checking for the file `/tdo-team/grpn/healthcheck`:

| Probe | Type | Command | Initial Delay | Period |
|-------|------|---------|---------------|--------|
| Readiness | exec | `cat /tdo-team/grpn/healthcheck` | 20s | 5s |
| Liveness | exec | `cat /tdo-team/grpn/healthcheck` | 30s | 15s |

Cronjob completion is signaled by each script touching `/tmp/signals/terminated` (`TERM_FILE`), which Kubernetes uses to detect clean job completion.
