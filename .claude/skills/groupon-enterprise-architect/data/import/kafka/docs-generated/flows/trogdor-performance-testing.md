# Trogdor Performance Testing

## Summary

Trogdor is Kafka's built-in distributed testing and fault-injection framework. An operator submits a workload task (produce benchmark, consume benchmark, network partition, process kill, etc.) to the Coordinator via REST API. The Coordinator's scheduler assigns the task to one or more Agents. Each Agent executes the task — typically producing or consuming traffic against `continuumKafkaBroker` — and reports progress back to the Coordinator. Results are available via the Coordinator's REST API.

## Trigger

- **Type**: manual
- **Source**: Operator or CI pipeline calls `POST /coordinator/task` on `kafkaTrogdorCoordinatorApi`
- **Frequency**: On-demand (performance testing cycles or pre-release validation)

## Participants

| Participant | Role in Flow | Architecture Ref |
|-------------|-------------|-----------------|
| Operator / CI Pipeline | Submits task definition and polls for results | External |
| Coordinator API | Accepts task submission and status query requests | `kafkaTrogdorCoordinatorApi` |
| Task Scheduler | Selects agent nodes and dispatches the task definition | `kafkaTrogdorCoordinatorScheduler` |
| Trogdor Coordinator | Orchestrates the full test lifecycle | `continuumKafkaTrogdorCoordinator` |
| Task Runner | Receives the task, executes the workload, and reports status | `kafkaTrogdorAgentTaskRunner` |
| Platform Adapter | Provides access to local host resources needed for fault-injection tasks | `kafkaTrogdorAgentPlatformAdapter` |
| Trogdor Agent | Executes the workload on its host node | `continuumKafkaTrogdorAgent` |
| Network API (Broker) | Receives produce/consume traffic generated by the agent during benchmark tasks | `kafkaBrokerNetworkApi` |

## Steps

1. **Operator submits task**: Operator sends a `POST /coordinator/task` request to `kafkaTrogdorCoordinatorApi` with a task specification (task type, target broker addresses, duration, throughput target, partition count, etc.)
   - From: `Operator / CI Pipeline`
   - To: `kafkaTrogdorCoordinatorApi`
   - Protocol: REST (HTTP/JSON)

2. **Coordinator accepts and validates task**: `kafkaTrogdorCoordinatorApi` validates the task spec and registers it in the coordinator's task registry
   - From: `kafkaTrogdorCoordinatorApi`
   - To: `kafkaTrogdorCoordinatorScheduler`
   - Protocol: direct (in-process)

3. **Scheduler selects target agents**: `kafkaTrogdorCoordinatorScheduler` selects one or more agent nodes based on the task spec (e.g., all agents, or a specific subset); computes per-agent subtask assignments
   - From: `kafkaTrogdorCoordinatorScheduler` (internal)
   - To: `kafkaTrogdorCoordinatorScheduler` (internal)
   - Protocol: direct (in-process)

4. **Dispatcher sends task to agents**: Coordinator sends a `POST /agent/task` request to each selected `continuumKafkaTrogdorAgent` with the subtask definition
   - From: `kafkaTrogdorCoordinatorScheduler`
   - To: `kafkaTrogdorAgentTaskRunner`
   - Protocol: REST (HTTP/JSON)

5. **Agent starts task runner**: `kafkaTrogdorAgentTaskRunner` receives the task and instantiates the appropriate workload implementation (e.g., `ProduceBenchSpec`, `ConsumeBenchSpec`, `NetworkPartitionFaultSpec`)
   - From: `kafkaTrogdorAgentTaskRunner` (internal)
   - To: `kafkaTrogdorAgentPlatformAdapter` (for fault tasks) or broker (for benchmark tasks)
   - Protocol: direct (in-process)

6. **Executes benchmark traffic against broker**: For produce/consume benchmark tasks, Task Runner uses an embedded Kafka producer or consumer to generate traffic against `kafkaBrokerNetworkApi` at the configured throughput
   - From: `kafkaTrogdorAgentTaskRunner`
   - To: `kafkaBrokerNetworkApi`
   - Protocol: Kafka Wire Protocol (TCP)

7. **Platform Adapter executes fault injection** (fault tasks only): For fault tasks (network partition, process kill, disk failure simulation), `kafkaTrogdorAgentPlatformAdapter` interacts with the host OS via process control, iptables, or filesystem operations
   - From: `kafkaTrogdorAgentTaskRunner`
   - To: `kafkaTrogdorAgentPlatformAdapter`
   - Protocol: direct (in-process)

8. **Agent reports task status**: Agent sends status updates to the Coordinator via `GET /agent/task/{id}` polling (or the Coordinator polls the agent) with current state: `RUNNING`, `DONE`, or `FAILED`
   - From: `kafkaTrogdorAgentTaskRunner`
   - To: `kafkaTrogdorCoordinatorApi` (via coordinator polling agent)
   - Protocol: REST (HTTP/JSON)

9. **Operator polls for results**: Operator calls `GET /coordinator/task/{id}` to retrieve the final task status, throughput metrics, and error summary
   - From: `Operator / CI Pipeline`
   - To: `kafkaTrogdorCoordinatorApi`
   - Protocol: REST (HTTP/JSON)

## Error Handling

| Error Scenario | Handling Strategy | Outcome |
|---------------|-------------------|---------|
| Agent unreachable | Coordinator marks the subtask as `FAILED`; overall task enters `FAILED` state | Operator must verify agent pod health and resubmit |
| Broker rejects produce (quota, wrong leader) | Task Runner logs produce errors; task completes with error count in results | Results indicate degraded throughput; useful for baseline measurement under failure conditions |
| Task timeout | Coordinator enforces `maxDurationMs` from the task spec; cancels the task if exceeded | Agent receives `DELETE /agent/task/{id}`; workload stops cleanly |
| Platform Adapter fault injection fails | Agent logs the failure; fault task enters `FAILED` state | Coordinator reports failure; operator investigates host permissions |
