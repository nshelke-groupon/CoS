# Data Stores

## Overview

Deal Alerts uses a single PostgreSQL database (`deal_alerts` schema) as its authoritative data store. The database holds deal snapshots, time-partitioned field deltas, alert lifecycle state, notification records, action tracking, severity configuration, and authentication tables. There are no separate caches; all reads go directly to PostgreSQL via Kysely with a connection pool of up to 10 connections.

#### Access Patterns

- **Read**: The web app reads via Kysely (typed SQL) with a pool of 10 connections, 30s idle timeout, and 5s connection timeout. Full-text search uses PostgreSQL tsvector/tsquery with weighted ranking. UUID lookups use direct equality predicates. Pagination is offset-based.
- **Write**: n8n workflows write via JDBC/SQL: bulk snapshot upserts use advisory locks (`pg_advisory_xact_lock`) per deal to prevent concurrent writes. Alert generation happens in-database via PL/pgSQL functions invoked within the `insert_snapshots` procedure. The web app writes admin operations (templates, action maps, muted alerts, monitored fields, severity matrices).
- **Indexes**: Full-text search via GIN index on `deal_search.search_vector`. Hash-based deduplication via `body_hash` on `deal_snapshots`. Monthly range partitioning on `deal_deltas` by timestamp with `ensure_partitions_ahead()` function.

## Caches

> No dedicated cache layer. All data is served directly from PostgreSQL.

## Data Flows

1. **MDS Ingestion**: n8n Deal Snapshots Ingestor pages MDS API -> calls `insert_snapshots()` PL/pgSQL function -> atomically upserts `deal_snapshots`, computes `deal_deltas`, and generates `alerts` via `generate_alerts_from_snapshot()`.
2. **External Alerts**: n8n External Alerts Importer pulls from BigQuery -> inserts into `alerts` table with `source='bigquery'`.
3. **Search Index**: `refresh_search_index()` function incrementally rebuilds `deal_search` from `deal_snapshots` when `body_hash` changes.
4. **Action Orchestration**: n8n Execute Actions reads pending alerts -> enriches with `alert_action_data` -> creates `actions` entries. Execute Alert Actions creates `salesforce_task_actions`/`chat_actions` and resolves alert status.
5. **Notification Pipeline**: SoldOut Notifications creates `notifications` records -> SMS Sender sends via Twilio and updates `notification_status_history` -> inbound replies populate `notification_replies`.
6. **Attribution**: Attributions workflow correlates `deal_deltas` replenishment events with `salesforce_tasks` and `notification_replies` to produce `alert_inventory_attributions`.
