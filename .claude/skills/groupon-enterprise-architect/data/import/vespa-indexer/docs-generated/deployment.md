# Deployment

## Overview

Vespa Indexer is containerised with Docker (`python:3.12-slim` base) and deployed to GCP-hosted Kubernetes clusters via Helm charts (`cmf-generic-api` 3.95.1 for the main service, `cmf-generic-cron-job` 3.95.1 for CronJobs). Deployments are managed by DeployBot (`deploy_kubernetes:v2.8.5`) using Helm templates rendered by `krane`. Two environments are active: staging (GCP stable US Central1) and production (GCP production US Central1). A staging-europe-west1 target exists in `.deploy_bot.yml` but is commented out.

## Infrastructure

| Component | Technology | Details |
|-----------|-----------|---------|
| Container | Docker (`python:3.12-slim`) | `Dockerfile` in repo root; exposes port 8080 |
| Orchestration | Kubernetes (GCP) | Helm chart `cmf-generic-api` 3.95.1; manifests generated by `.meta/deployment/cloud/scripts/deploy.sh` |
| CronJobs | Kubernetes CronJob | Helm chart `cmf-generic-cron-job` 3.95.1; alpine 3.18 container running `curl` to trigger endpoints |

| Metrics | Prometheus sidecar | `prometheus:v2.51.2`; scrape interval 60s; remote write to Thanos (`thanos-receive.telegraf-<env>.svc:19291`) |
| Image registry | `docker-conveyor.groupondev.com/relevance/vespa-indexer` | Tagged by commit SHA |

## CI/CD Pipeline

- **Tool**: Jenkins
- **Config**: `Jenkinsfile` (repo root)
- **Trigger**: On push to `master`, `main`, `release`, `feature` branches

### Pipeline Stages

1. **Build**: Docker image built with `dockerBuildPipeline` shared library; git submodules cloned (`cloneSubModules: true`)
2. **Test**: Executes `./run_tests.sh` (runs `pytest` with coverage)
3. **Push**: Docker image pushed to `docker-conveyor.groupondev.com/relevance/vespa-indexer`
4. **Deploy (staging)**: Automatic promotion to `staging-us-central1` on successful build
5. **Deploy (production)**: Manual promotion from staging to `production-us-central1` via DeployBot (`promote_to: production-us-central1`)

### Deploy Script

`.meta/deployment/cloud/scripts/deploy.sh` renders and applies Kubernetes manifests using `krane` with a 300-second global timeout. It deploys:
1. Main service (`cmf-generic-api`)
2. Prometheus sidecar
3. Deal-refresh CronJob (`cmf-generic-cron-job`)
4. Feature-refresh CronJob (`cmf-generic-cron-job`)

## CronJob Schedules

| CronJob | Schedule (UTC) | Timeout | Retry limit | Concurrency |
|---------|----------------|---------|-------------|-------------|
| `cronjob-deal-refresh` | Daily at 10:00 (`0 10 * * *`) | 2 hours | 2 | Forbid |
| `cronjob-feature-refresh` | 04:00, 16:00, 22:00 (`0 4,16,22 * * *`) | 2 hours | 2 | Forbid |

CronJobs run an Alpine 3.18 container that issues a `curl -X POST` to the appropriate scheduler endpoint on the main service (`${VESPA_SERVICE_URL}/scheduler/refresh-deals` or `/scheduler/refresh-features`). `cronjobConcurrencyPolicy: Forbid` prevents concurrent runs.
