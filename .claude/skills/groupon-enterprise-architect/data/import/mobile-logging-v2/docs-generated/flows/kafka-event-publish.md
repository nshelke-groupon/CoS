# Kafka Event Publish

## Summary

This flow describes how an encoded GRP event is submitted to Kafka after normalisation and serialisation. The Kafka Publish Component selects the appropriate Kafka producer (primary or EMEA), calls the producer's `send()` method with the encoded payload targeting the `mobile_tracking` topic, handles the acknowledgement or failure, and emits outcome metrics. The Kafka connection is established over TLS using a JKS keystore generated at container startup.

## Trigger

- **Type**: event (internal)
- **Source**: `mobileLogV2_encodingComponent` delivers an encoded payload for each normalised GRP event
- **Frequency**: Once per valid decoded event within a log upload request

## Participants

| Participant | Role in Flow | Architecture Ref |
|-------------|-------------|-----------------|
| Event Encoding Component | Serialises the normalised event and hands off the payload | `mobileLogV2_encodingComponent` |
| Kafka Publish Component | Selects producer and calls Kafka `send()` | `mobileLogV2_kafkaPublishComponent` |
| Kafka Message Bus | Receives and stores the event on the `mobile_tracking` topic | `messageBus` |
| Metrics Stack | Records publish outcome (success/error) per event type | `metricsStack` |
| Logging Stack | Receives `KAFKA_PRODUCER_SENT` or `KAFKA_SEND` error log entries | `loggingStack` |

## Steps

1. **Receive encoded payload**: The `mobileLogV2_kafkaPublishComponent` receives the serialised event bytes produced by `mobileLogV2_encodingComponent` via the internal RxJava pipeline.
   - From: `mobileLogV2_encodingComponent`
   - To: `mobileLogV2_kafkaPublishComponent`
   - Protocol: Direct (in-process)

2. **Select Kafka producer**: The `KafkaEventsProducerFactory` determines which producer to use. The primary `kafkaProducer` is used for all events; the `kafkaEmeaProducer` is available for EMEA-specific routing as configured.
   - From: `mobileLogV2_kafkaPublishComponent` (internal)
   - Protocol: Direct

3. **Apply producer configuration**: The selected Kafka producer is configured with:
   - Bootstrap broker: from `kafkaProducer.broker` config
   - Topic: from `kafkaProducer.topic` config (e.g., `mobile_tracking`)
   - Client ID: `kafkaProducer.clientId` + thread ID
   - Acks: `1` (leader acknowledgement only)
   - Retries: `1` (one retry on transient failure)
   - Security protocol: `SSL` (if `kafkaConnectionConfig.useTLS == true`)
   - SSL keystore: `/var/groupon/jtier/kafka.client.keystore.jks` (generated by `kafka-tls.sh`)
   - Trust store: `/var/groupon/jtier/truststore.jks` (Groupon Root CA)
   - From/To: `mobileLogV2_kafkaPublishComponent` → Kafka client library
   - Protocol: TLS/SSL configuration

4. **Send event to Kafka**: The Kafka client calls `producer.send(record)` with the encoded payload. The `ProducerRecord` targets the configured topic.
   - From: `mobileLogV2_kafkaPublishComponent`
   - To: `messageBus` (Kafka broker, `mobile_tracking` topic)
   - Protocol: Kafka/TLS (port determined by broker config)

5. **Receive acknowledgement**: The Kafka broker acknowledges receipt with `acks=1` (leader acknowledgement). On success, a `KAFKA_PRODUCER_SENT` log event is emitted.
   - From: `messageBus`
   - To: `mobileLogV2_kafkaPublishComponent`
   - Protocol: Kafka

6. **Emit publish metrics**: A `success` or `error` outcome metric is emitted per event type to the metrics stack.
   - From: `mobileLogV2_kafkaPublishComponent`
   - To: `metricsStack`
   - Protocol: tsdaggr

7. **Log outcome**: A `KAFKA_SEND` log event (success or error) is written to the logging stack.
   - From: `continuumMobileLoggingService`
   - To: `loggingStack`
   - Protocol: Filebeat log shipping

## Error Handling

| Error Scenario | Handling Strategy | Outcome |
|---------------|-------------------|---------|
| Kafka broker unreachable | Kafka client retries once (`retries=1`); logs `KAFKA_SEND` error; emits `error` metric | Event is dropped; no local persistence |
| TLS handshake failure | Kafka client raises `SslAuthenticationException`; one retry attempted | Event is dropped; check keystore generation in pod logs |
| Ack timeout | Kafka client retries once; raises exception on second failure | Event is dropped; logged as error |
| Message too large | Kafka client raises `RecordTooLargeException` | Event is dropped; logged as exception |

## Sequence Diagram

```
mobileLogV2_encodingComponent -> mobileLogV2_kafkaPublishComponent: Deliver encoded event payload
mobileLogV2_kafkaPublishComponent -> mobileLogV2_kafkaPublishComponent: Select producer (primary or EMEA)
mobileLogV2_kafkaPublishComponent -> messageBus: producer.send(ProducerRecord[mobile_tracking, payload]) over TLS
messageBus --> mobileLogV2_kafkaPublishComponent: Ack (acks=1, leader)
mobileLogV2_kafkaPublishComponent -> metricsStack: Emit success counter (per event type)
mobileLogV2_kafkaPublishComponent -> loggingStack: Log KAFKA_PRODUCER_SENT
```
