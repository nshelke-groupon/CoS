# Data Stores

## Overview

The NetSuite customizations operate exclusively within Oracle NetSuite's managed cloud platform. All transactional data (vendor bills, purchase orders, vendor payments, journal entries, custom records) resides in NetSuite's internal Oracle database, accessed only through SuiteScript API calls (`nlapiCreateRecord`, `nlapiSearchRecord`, `nlapiSubmitRecord`, `nlapiLoadRecord`). No external database connection strings or credentials are used. The File Cabinet (NetSuite's managed file storage) serves as a staging area for JSON export files generated by scheduled scripts before they are transmitted to downstream systems.

#### Access Patterns

- **Read**: Scheduled scripts query records via `nlapiCreateSearch` / `nlapiSearchRecord` with date-range filters (`lastmodifieddate onorafter`) to detect changes since last run
- **Write**: RESTlets create new `vendorbill` and `vendorcredit` records via `nlapiCreateRecord` + `nlapiSubmitRecord`; User Event scripts update fields on save via `nlapiSubmitField` and `nlapiSubmitRecord`
- **Indexes**: NetSuite manages all indexing internally; searches exploit standard indexed fields (`internalid`, `lastmodifieddate`, `tranid`, `subsidiary`, `entity`)

#### Key Entities

| Entity / Table | Purpose | Key Fields |
|----------------|---------|-----------|
| JSON export files | Serialized `purchaseOrders` JSON written by OTP scheduled scripts | Named as `<timestamp>_GLS_otp_json.txt` or `<timestamp>_POmgr_EMEA_otp_json.txt` |

#### Access Patterns

- **Write**: Scheduled scripts create files via `nlapiCreateFile` and persist them with `nlapiSubmitFile`
- **Read**: JPM Suitelet queries File Cabinet folder contents via `nlapiSearchRecord('file', ...)` to present pending payment files to users

## Caches

> No evidence found in codebase. NetSuite does not expose an application-level cache. All data access goes through the SuiteScript API to the NetSuite database.

## Data Flows

1. **OTP export**: Scheduled scripts query modified purchase orders from the NetSuite ERP database, serialize them to JSON, write the file to the File Cabinet, then POST the JSON body to the downstream system (GLS or PO Manager). On success, the `customrecord_poes_auto_ctrl` last-success timestamp is updated.
2. **Vendor bill creation**: External systems (GLS via SnapLogic) POST invoice data to the vendor bill RESTlet, which creates `vendorbill` or `vendorcredit` records in the NetSuite ERP database and returns NetSuite internal IDs.
3. **JPM payment batch**: Finance staff submit JPM ACH files staged in the File Cabinet via the JPM Suitelet, which triggers a SnapLogic pipeline that picks up those files and transmits them to JPMorgan Chase.
