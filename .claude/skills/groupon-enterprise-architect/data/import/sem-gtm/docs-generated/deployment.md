# Deployment

## Overview

`sem-gtm` is deployed as two independent Kubernetes Deployments (tagging and preview) onto GCP-hosted GKE clusters. Docker images are built via Jenkins using a Google-managed base image (`gcr.io/cloud-tagging-10302018/gtm-cloud-image:stable`) and published to Groupon's internal Docker registry. Deployments are executed by Deploybot, which calls a Helm + Kustomize + krane pipeline. Both sub-services deploy to the same namespaces but with independent scaling and configuration.

## Infrastructure

| Component | Technology | Details |
|-----------|-----------|---------|
| Container image | Docker | `Dockerfile` (single line: `FROM gcr.io/cloud-tagging-10302018/gtm-cloud-image:stable`) |
| Image registry | docker-conveyor.groupondev.com | `docker-conveyor.groupondev.com/transam/sem-gtm` |
| Orchestration | Kubernetes (GKE) | Manifests generated by Helm chart `cmf-generic-api` v3.89.0 + Kustomize overlays |

| Load balancer | GCP Conveyor (VIP) | Per-environment VIP endpoints defined in production overlay YAMLs |

## CI/CD Pipeline

- **Tool**: Jenkins + GitHub Actions (PR title check only)
- **Config**: `Jenkinsfile` (build), `.github/workflows/pr-title-check.yml` (PR validation), `.deploy_bot.yml` (deploy targets)
- **Trigger**: Push to `main` branch triggers build and deploy to `staging-us-central1`; promotion to `production-us-central1` follows successful staging deployment

### Pipeline Stages

1. **PR Title Check**: GitHub Actions validates PR title matches `[JIRA-KEY-123] Description` format
2. **Docker Build**: Jenkins (`java-pipeline-dsl@latest-2`) builds the Docker image from `Dockerfile` and pushes to `docker-conveyor.groupondev.com/transam/sem-gtm`
3. **Deploy to Staging**: Deploybot executes `bash ./.meta/deployment/cloud/scripts/deploy.sh staging-us-central1 staging sem-gtm-staging` — runs Helm template for both preview and tagging components, applies Kustomize overlays, deploys with krane
4. **Promote to Production**: Deploybot promotes staging artifact to production via `deploy.sh production-us-central1 production sem-gtm-production`

## Health Probes

Both sub-services use identical probe configuration:

| Probe | Path | Initial Delay | Period | Timeout |
|-------|------|--------------|--------|---------|
| Readiness | `/healthz` | 300s | 30s | 4s |
| Liveness | `/healthz` | 300s | 30s | 4s |

> The 300-second initial delay accounts for GTM Cloud image startup time while it fetches container configuration from Google's platform.
