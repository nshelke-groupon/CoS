# Integrations

## Overview

Cloudability has three external dependencies (Cloudability Provisioning API, Cloudability Ingestion API, and the Kubernetes API Server) and one internal dependency (the Conveyor Kubernetes cluster infrastructure). The provisioning CLI is operator-run and calls the Cloudability SaaS API directly. The deployed Metrics Agent autonomously interacts with the Kubernetes API Server and Cloudability's ingestion endpoint.

## External Dependencies

| System | Protocol | Purpose | Critical | Architecture Ref |
|--------|----------|---------|----------|-----------------|
| Cloudability Provisioning API | REST/HTTPS | Registers clusters and fetches generated agent manifests | yes | `cloudabilityProvisioningApi_1be936` |
| Cloudability Ingestion API | HTTPS | Receives uploaded Kubernetes metric samples from the agent | yes | `cloudabilityIngestionApi_c07c76` |
| Kubernetes API Server | Kubernetes API | Provides cluster, node, pod, and workload metrics to the agent | yes | `kubernetesApiServer_272a31` |

### Cloudability Provisioning API Detail

- **Protocol**: HTTPS/REST
- **Base URL**: `https://api.cloudability.com/v3/containers/provisioning`
- **Auth**: HTTP Basic Auth — per-user or service API key stored in `secrets/provisioning-api-key`
- **Purpose**: Registers Conveyor Kubernetes clusters as monitored environments; returns generated Kubernetes manifests for agent deployment
- **Failure mode**: Cluster registration and manifest fetch cannot proceed; agent cannot be deployed to new clusters
- **Circuit breaker**: No evidence found in codebase

### Cloudability Ingestion API Detail

- **Protocol**: HTTPS
- **Base URL**: Configured within the agent manifest generated by the Provisioning API
- **Auth**: Cluster-scoped token embedded in the generated manifest (stored in `secrets/`)
- **Purpose**: Receives periodic metric sample uploads (node, pod, namespace, workload usage) from the running Metrics Agent
- **Failure mode**: Metric data gaps appear in the Cloudability portal; data may be delayed up to 24 hours before alert conditions are visible
- **Circuit breaker**: No evidence found in codebase (governed by third-party agent runtime)

### Kubernetes API Server Detail

- **Protocol**: Kubernetes API (HTTPS + watch/list verbs)
- **Base URL**: In-cluster API server for each Conveyor Kubernetes cluster
- **Auth**: Kubernetes ServiceAccount `cloudability` bound to the `generic-metrics-agent` ClusterRole via `cloudability-metrics-agent` ClusterRoleBinding
- **Purpose**: Provides live cluster state — namespaces, nodes, pods, replication controllers, services, deployments, replica sets, daemon sets, jobs, cron jobs, persistent volumes, persistent volume claims, and proxy/stats/metrics endpoints
- **Failure mode**: Agent cannot collect metrics; pod restarts with errors in logs
- **Circuit breaker**: No evidence found in codebase

## Consumed By

> Upstream consumers are tracked in the central architecture model. The Cloudability portal (`https://app.cloudability.com/#/containers`) is the primary consumer of data produced by this service. Engineering teams access cost data through the portal; access is provisioned via ARQ/Launchpad.

## Dependency Health

- The Cloudability portal data is delayed by approximately 24 hours. Changes should be verified after 24 hours.
- Monitor pod status with `kubectl -n cloudability-staging get pods` (or `cloudability-production`). Restarts above 0 indicate an issue.
- Check the [Cloudability Status Page](https://status.cloudability.com/) for upstream SaaS outages.
- Wavefront metrics dashboard: `https://groupon.wavefront.com/u/KRNCKsVKGz?t=groupon`
- Verify the `generic-metrics-agent` ClusterRole and `cloudability-metrics-agent` ClusterRoleBinding exist on target clusters before deploying.
