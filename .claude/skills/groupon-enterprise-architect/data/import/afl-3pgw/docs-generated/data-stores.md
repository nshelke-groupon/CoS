# Data Stores

## Overview

AFL-3PGW owns a single MySQL database (`afl_3pgw_production`) that serves as the authoritative store for all affiliate order submission state. The database holds attribution data imported from upstream systems (via Optimus Prime jobs), correction records generated by reconciliation logic, job execution checkpoints, CJ report data, Awin transaction records, and audit trails of every outbound submission. There are no caches or secondary stores.

#### Key Entities

| Entity / Table | Purpose | Key Fields |
|----------------|---------|-----------|
| `audit_cj_submitted_orders` | Audit trail of all orders submitted to Commission Junction in real time | order ID, CJ event, submission timestamp, status |
| `cj_cancellation` | Stores cancellation, refund, and chargeback records destined for CJ correction submissions | order ID, correction type, region, processed flag |
| `awin_cancellation` | Stores Awin-bound correction and cancellation records | order ID, correction type, advertiser ID, processed flag |
| `correction_submission` | Tracks correction batches generated by the Reconciliation Job and their submission status | correction ID, type, checkpoint, submitted_at |
| `jobs` | Quartz job execution checkpoints; records last-run timestamps used by scheduled jobs to determine processing windows | job name, last_run, next_run |
| `ma_attribution_v2` | Imported MA (marketing attribution) data from Teradata/Optimus Prime; source data for reconciliation | order ID, attribution ID, region, partner |
| `network_report` | CJ and Awin network report data fetched by report processing jobs | report date, network, line items |
| `order_correction` | Individual order-level corrections (new_sale, refund, cancelled, chargeback) queued for submission | order ID, correction_type, amount, currency, status |

#### Access Patterns

- **Read**: Reconciliation and submission jobs query corrections by checkpoint timestamp and region; report jobs read from network report tables; MBUS handler reads attribution state before submitting
- **Write**: Real-time MBUS handler writes audit records after each CJ/Awin submission; Quartz jobs write correction records, update checkpoints, and insert report rows; Optimus Prime import jobs bulk-insert MA attribution data
- **Indexes**: Not directly visible in this inventory; evidence of indexed lookups on order ID, region, job name, and processed/status flags based on DAO query patterns

## Caches

> This service does not use any explicit cache layer. All reads and writes go directly to MySQL.

## Data Flows

- **Optimus Prime import**: External Optimus jobs (`AFL-3PGW-IMPORT-MA-ATTRIBUTIONS-ORDERS`, `AFL-3PGW-IMPORT-CJ-COMMISSIONS`, `AFL-3PGW-IMPORT-MA-CANCELLED-ORDERS`) bulk-load data from Teradata into the MySQL database. These are prerequisite steps for the reconciliation pipeline.
- **Reconciliation pipeline**: `ReconciliationJob` reads imported MA data, computes corrections (new_sale, refund, cancelled, chargeback), and writes records into the `order_correction` / `correction_submission` tables.
- **Submission pipeline**: `CjSubmitNewOrdersJob` and `CjCorrectionSubmissionJob` read from correction tables, submit to CJ, and update submission status and job checkpoints.
- **Real-time audit**: `OrdersMessageHandler` writes to `audit_cj_submitted_orders` after each real-time CJ or Awin submission.
- **Report pipeline**: `CjReportProcessingJob` and `CjCommissionsJob` fetch CJ report data and write into `network_report` and related tables.
