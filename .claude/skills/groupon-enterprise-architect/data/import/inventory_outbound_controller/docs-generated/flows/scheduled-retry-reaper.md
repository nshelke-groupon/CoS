# Scheduled Retry Reaper

## Summary

The Scheduled Retry Reaper flow encompasses two related Quartz-scheduled maintenance jobs within `outboundSchedulingJobs`: a retry job that reprocesses fulfillment records that previously failed or were left in an intermediate state, and a scavenger (reaper) job that identifies and reaps expired or permanently stale fulfillment records. Together these jobs ensure that transient failures in fulfillment dispatch, inventory eligibility checks, or external service calls do not permanently block order processing. Both jobs operate against `continuumInventoryOutboundControllerDb` and may re-trigger downstream integrations as part of the retry cycle.

## Trigger

- **Type**: schedule
- **Source**: Quartz 2.3.0 scheduler within `outboundSchedulingJobs`; scavenger also triggerable on demand via `POST /start-scavenger`; jobs schedulable via `POST /admin/jobs/schedule`
- **Frequency**: Periodic intervals configured in `conf/application.conf` (exact cron expressions not confirmed from inventory)

## Participants

| Participant | Role in Flow | Architecture Ref |
|-------------|-------------|-----------------|
| Goods Outbound Controller | Executes retry and reaper job logic via `outboundSchedulingJobs` and `outboundFulfillmentOrchestration` | `continuumInventoryOutboundController` |
| Outbound Controller DB | Source of failed/stale fulfillment records; updated as retry or reap actions complete | `continuumInventoryOutboundControllerDb` |
| Inventory Service | Re-queried during retry to recheck inventory eligibility | `continuumInventoryService` |
| Goods Inventory Service | Re-queried during retry to recheck goods-level routing data | `continuumGoodsInventoryService` |
| Orders Service | Re-queried during retry to recheck order state | `continuumOrdersService` |
| Message Bus | Receives any sales order update events generated by successful retries | `messageBus` |

## Steps

### Retry Job

1. **Quartz fires retry job**: `outboundSchedulingJobs` triggers the retry job on its configured schedule.
   - From: Quartz scheduler (internal to `continuumInventoryOutboundController`)
   - To: `outboundFulfillmentOrchestration`
   - Protocol: direct (in-process)

2. **Query failed fulfillment records**: `outboundPersistenceAdapters` reads fulfillment records in failed, pending, or retry-eligible states from `continuumInventoryOutboundControllerDb`.
   - From: `continuumInventoryOutboundController`
   - To: `continuumInventoryOutboundControllerDb`
   - Protocol: JDBC / MySQL

3. **Recheck inventory eligibility**: For each failed record, `outboundExternalServiceClients` re-queries Inventory Service and Goods Inventory Service to determine whether the conditions that caused the failure have been resolved.
   - From: `continuumInventoryOutboundController`
   - To: `continuumInventoryService`, `continuumGoodsInventoryService`
   - Protocol: HTTP / REST

4. **Recheck order state**: `outboundExternalServiceClients` re-queries Orders Service to confirm the order remains in a fulfillable state (not cancelled or completed by another path).
   - From: `continuumInventoryOutboundController`
   - To: `continuumOrdersService`
   - Protocol: HTTP / REST

5. **Re-attempt fulfillment dispatch**: `outboundFulfillmentOrchestration` re-dispatches the fulfillment (e.g., re-sends to Landmark Global 3PL or re-routes inventory) for eligible records.
   - From: `continuumInventoryOutboundController`
   - To: external 3PL (`extLandmarkGlobalApi_0f3b5de8`) or internal routing
   - Protocol: HTTPS / HTTP

6. **Update fulfillment status**: `outboundPersistenceAdapters` updates the record status in MySQL to reflect retry success, ongoing retry, or failure escalation.
   - From: `continuumInventoryOutboundController`
   - To: `continuumInventoryOutboundControllerDb`
   - Protocol: JDBC / MySQL

7. **Publish sales order update** (on successful retry): `outboundMessagingAdapters` publishes a `jms.topic.goods.salesorder.update` event when a fulfillment is successfully retried.
   - From: `continuumInventoryOutboundController`
   - To: `messageBus`
   - Protocol: JMS

### Scavenger (Reaper) Job

1. **Quartz fires scavenger job** (or manual trigger via `POST /start-scavenger`): `outboundSchedulingJobs` triggers the scavenger on schedule or on demand.
   - From: Quartz scheduler or admin API caller
   - To: `outboundFulfillmentOrchestration`
   - Protocol: direct (in-process) or HTTP

2. **Query expired / stale fulfillment records**: `outboundPersistenceAdapters` reads fulfillment records that have exceeded their expiry threshold or are permanently unresolvable.
   - From: `continuumInventoryOutboundController`
   - To: `continuumInventoryOutboundControllerDb`
   - Protocol: JDBC / MySQL

3. **Mark records as reaped / terminal**: `outboundFulfillmentOrchestration` transitions stale records to a terminal state (e.g., expired, abandoned) in MySQL.
   - From: `continuumInventoryOutboundController`
   - To: `continuumInventoryOutboundControllerDb`
   - Protocol: JDBC / MySQL

## Error Handling

| Error Scenario | Handling Strategy | Outcome |
|---------------|-------------------|---------|
| Inventory Service unavailable during retry | HTTP call fails; record remains in retry-eligible state | Retried on next scheduled run |
| Orders Service unavailable during retry | HTTP call fails; retry deferred | Retried on next scheduled run |
| 3PL dispatch fails again on retry | Failure recorded; retry counter incremented | Record escalated toward terminal state after max retries exceeded |
| MySQL write failure during status update | Transaction rolled back; record remains in current state | Job logs error; record will be picked up on next run |
| Scavenger reaps record prematurely | No safeguard evidenced | Fulfillment permanently abandoned; requires manual intervention |

## Sequence Diagram

```
Quartz -> outboundSchedulingJobs: fire retry job
outboundSchedulingJobs -> outboundFulfillmentOrchestration: executeRetry()
outboundFulfillmentOrchestration -> DB: SELECT fulfillments WHERE status IN (FAILED, PENDING_RETRY)
DB --> outboundFulfillmentOrchestration: failed fulfillment records
outboundFulfillmentOrchestration -> InventoryService: GET eligibility check
InventoryService --> outboundFulfillmentOrchestration: eligibility result
outboundFulfillmentOrchestration -> OrdersService: GET order state
OrdersService --> outboundFulfillmentOrchestration: order state
outboundFulfillmentOrchestration -> extLandmarkGlobalApi: POST fulfillment instructions (retry)
extLandmarkGlobalApi --> outboundFulfillmentOrchestration: dispatch result
outboundFulfillmentOrchestration -> DB: UPDATE fulfillment status
outboundFulfillmentOrchestration -> MessageBus: publish jms.topic.goods.salesorder.update (on success)
```
