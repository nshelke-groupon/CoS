# Integrations

## Overview

The plugin has four runtime integration points. Three are internal Groupon services (Lazlo GAPI, Orders service, and Groupon MBus) and one is the Kill Bill platform itself (in-process). The plugin is a pure consumer — it makes outbound calls to create orders and read order/payment status, and it subscribes to events; it does not serve as a dependency for other named services.

## External Dependencies

| System | Protocol | Purpose | Critical | Architecture Ref |
|--------|----------|---------|----------|-----------------|
| Groupon MBus | JMS/STOMP | Consume `TransactionalLedgerEvents` for payment reconciliation | yes | `messageBus` |

### Groupon MBus Detail

- **Protocol**: JMS/STOMP via `mbus-client` 1.2.16
- **Base URL / SDK**: STOMP host configured per-tenant and per-region (e.g., `mbus-stg-na.us-central1.mbus.stable.gcp.groupondev.com:61613`)
- **Auth**: Per-tenant MBus `userName` and `password` from plugin tenant configuration (`message-bus` YAML section)
- **Purpose**: Receives `TransactionalLedgerEvents` from the Orders system. Events trigger the plugin to call the Orders service and reconcile the corresponding Kill Bill invoice payment state (mark as paid, record refunds, chargebacks).
- **Failure mode**: Failed message processing is retried via the Kill Bill notification queue (`sp_notifications`). MBus listener thread health is reflected in the plugin healthcheck.
- **Circuit breaker**: No evidence found in codebase.

## Internal Dependencies

| Service | Protocol | Purpose | Architecture Ref |
|---------|----------|---------|-----------------|
| Lazlo GAPI (`api-lazlo`) | HTTPS/JSON | Creates subscription orders for invoiced amounts via `v2/users/{userId}/multi_item_orders` | `continuumApiLazloService` |
| Orders Service | HTTPS/JSON (Retrofit2) | Reads order and payment status to reconcile Kill Bill invoice state after ledger events | `continuumOrdersService` |
| Kill Bill Platform | In-process OSGi API | Delivers `INVOICE_CREATION` events; provides account, subscription, invoice, custom field, and tag APIs | Kill Bill OSGi runtime |

### Lazlo GAPI (`continuumApiLazloService`) Detail

- **Protocol**: HTTPS/JSON (async HTTP via `com.ning:async-http-client`)
- **Base URL**: Per-tenant and per-region; configured as `gapi.region.<region>.url` in plugin tenant config (e.g., `https://api-staging.internal.snc1`)
- **Endpoint called**: `POST /v2/users/{userId}/multi_item_orders?client_id={clientId}`
- **Auth**: `X-SP-Auth-Token` header (short-lived token generated by `TokenManager`); `X-AutoRebuy-Auth-Token` header also sent for legacy compatibility; `client_id` query param from `gapi.clientId` config
- **Purpose**: Creates the actual Groupon commerce order for a billing period. Returns an `orderId` which is stored as an `ORDER_ID` custom field on the Kill Bill invoice.
- **Failure mode**: On failure, the order error details are stored as `ORDER_FAILED` on the invoice. The plugin does not automatically retry GAPI failures (retry must be triggered manually via `POST /plugins/sp-plugin/orders/v1`).
- **Circuit breaker**: No evidence found in codebase.

### Orders Service (`continuumOrdersService`) Detail

- **Protocol**: HTTPS/JSON via Retrofit2
- **Base URL**: Per-tenant and per-region; configured as `orders.region.<region>.url` in plugin tenant config (e.g., `http://orders-read-proxy1-staging.snc1`)
- **Auth**: No auth configuration evidence beyond URL; called with consumer UUID as the external key
- **Purpose**: Reads existing orders and payment status for a Groupon user. Used during ledger event processing to refresh Kill Bill invoice state and also during subscription creation to locate the billing record ZIP code.
- **Failure mode**: Errors are logged (Splunk step `syncOrder.callOrders.failure`). Retry is scheduled via notification queue.
- **Circuit breaker**: No evidence found in codebase.

## Consumed By

> Upstream consumers are tracked in the central architecture model. The plugin is embedded in the Kill Bill platform and is not directly called by other named services. External operators and internal automation trigger its HTTP endpoints for manual operations.

## Dependency Health

The plugin healthcheck at `GET /plugins/sp-plugin/healthcheck/cloud` reports healthy only when all configured MBus listener threads are running. Individual dependency health (GAPI, Orders) is monitored indirectly via Splunk alert searches on log steps `createOrder.callGAPI.failure` and `syncOrder.callOrders.failure`. No programmatic circuit breakers or retry policies are implemented beyond the Kill Bill notification queue retry mechanism.
